# TARA AI 平台开发计划书

本开发计划书基于系统设计文档 [system_design.md](file:///home/ubuntu/tara-ai-platform/docs/system_design.md) 制定，将开发流程划分为 6 个主要阶段。每个阶段包含具体的开发任务、依赖项和验收标准，以确保所有核心业务规则（BR-03 至 BR-78）的 100% 落地。

---

## 阶段 1：环境搭建与数据库初始化 (第 1 - 2 周)

### 任务细分
1.  **项目骨架与依赖配置**：
    *   构建 FastAPI 后端应用架构（目录结构分层：API 路由、服务层、数据访问层、模型层）。
    *   配置 PostgreSQL/SQLite 连接池，集成 Alembic 进行数据库表迁移管理。
    *   配置 Celery 及 Redis (作为消息代理和缓存媒介)。
2.  **核心数据库模型编写 (Database Models)**：
    *   实现 `users`, `projects`, `domains`, `diagrams`, `assets`, `tara_runs`, `tara_steps`, `system_settings` 的 SQLModel/SQLAlchemy 映射。
    *   设置完整的外键关联，特别是级联删除约束（如删除域控时自动清空相关分析记录和资产，BR-04/09）。
3.  **系统级备份与恢复工具开发 (BR-65/66)**：
    *   实现 `manage.py backup` 和 `manage.py restore` 命令行工具。
    *   实现一键打包 SQLite/PostgreSQL 数据和本地导出报告存储卷的 `.tar.gz` 功能，并在新容器中验证秒级还原。

### 阶段依赖
*   无

### 验收标准
*   [ ] 执行数据迁移脚本后，8 张数据库表成功建立且外键约束符合 ERD。
*   [ ] Celery Worker 与 Redis 连接通畅，测试异步任务投递与状态获取。
*   [ ] 运行 `backup` 脚本生成有效的压缩包，运行 `restore` 能将数据库恢复至备份前状态。

## 阶段 2：用户认证与基础管理 API (第 2 - 3 周)

### 任务细分
1.  **用户认证与两级角色权限限制 (BR-64)**：
    *   实现用户登录、密码的 `bcrypt` 加密与 JWT 生成校验。
    *   编写角色依赖注入拦截器（区分 `admin` 和 `analyst` 权限）。
2.  **项目与子域控管理 API (BR-03, BR-78, BR-10)**：
    *   实现项目创建（必填 50 字限制、选填 200 字限制）及模糊搜索。
    *   实现子域控创建、修改与进度查询 API。
    *   编写项目状态自动推导触发器 `recalculate_project_status`。
    *   编写中间件拦截器：
        *   对已归档（`completed`）项目拦截所有写入请求（返回 403，只读保护）。
        *   对处于“分析中”（`running`）状态的子域控拦截所有画布和资产的写入请求（返回 400）。

### 阶段依赖
*   阶段 1 数据库结构就绪。

### 验收标准
*   [ ] 模拟用户登录，JWT Token 能正确鉴权并限制非法跨域访问。
*   [ ] 当子域控分析状态从 `running` 改为 `completed` 时，项目状态能够按 BR-03 逻辑流转。
*   [ ] 归档只读拦截器运行正常，分析中域控锁定测试通过。

---

## 3. 阶段 3：DFD 画布编辑器与资产提取 (第 3 - 5 周)

### 任务细分
1.  **React Flow 画布集成与数据持久化 (BR-13, BR-16, BR-17)**：
    *   前端集成 React Flow 画布，设计符合 BR-13 定义的节点与连线数据属性。
    *   前端实现 2 秒防抖自动保存，并发起 `PUT /api/diagrams/:id` 保存 `snapshot_json` 和 `version_no`。
    *   实现乐观锁冲突检测（若后端返回 409，提示并引导备份本地 LocalStorage 数据以防覆写）。
    *   前端实现离线 LocalStorage 加密暂存及联网后自动同步逻辑。
2.  **基于 Redis 的独占式编辑锁与 WebSocket 广播 (BR-72)**：
    *   后端实现 `lock:domain:{id}` 的 Redis 心跳租约锁（5 分钟过期，前端通过 WebSocket 每 30 秒心跳续期）。
    *   实现锁定状态的 WebSocket 全局广播，当 User A 抢锁成功后，强制使 User B 的页面显示“用户[User A]正在编辑”并进入只读禁用状态。
3.  **资产自动提取与单域控 AI 去重 (BR-25, BR-29, BR-33, BR-35)**：
    *   编写 DFD Parser，解析节点和边属性，生成或更新 `assets` 表数据。
    *   **提取保留算法**：删除所有 `draft` 资产，保留所有已被用户标记为 `confirmed` 或 `rejected` 的资产。
    *   实现单域控内的 AI 资产去重管道（组装 Prompt 调用 LLM，解析返回的合并 ID，将被删除资产标记为 `rejected` 并记录合并原因）。

### 阶段依赖
*   阶段 2 的 API 和用户身份验证就绪。

### 验收标准
*   [ ] 双人同时保存同一个画布时，后提交者会触发 409 并弹出冲突提示。
*   [ ] 断开网络编辑画布 30 秒，重新联网后，本地暂存数据顺利同步回数据库。
*   [ ] 自动提取资产时，已由人工确认或拒绝的资产在其来源 DFD 删除或重提时不会丢失。
*   [ ] 执行 AI 去重，多余冗余资产状态变为 `rejected` 且备注记录合并链路。

## 阶段 4：异步 TARA 分析引擎与 Celery Worker (第 5 - 8 周)

### 任务细分
1.  **异步 Celery 跑批管道与启动校验 (BR-36, BR-37, BR-40, BR-41)**：
    *   实现启动 TARA 分析 API。检测子域控下是否至少有 1 个 `confirmed` 资产，否则拦截并报错。
    *   启动后，将子域控状态变更为 `running`，通过 Celery 后台派发 5 阶段串行分析任务。
2.  **增量分析与断点续跑哈希匹配 (BR-45)**：
    *   实现各步骤的 `input_hash` 计算算法（依赖输入资产属性和前置步骤结论）。
    *   在 Celery Task 单步执行时，对比前次运行的 `tara_steps`。若状态为 `completed` 且 Hash 一致，直接复制旧结果并跳过大模型调用。
3.  **人工修改继承与风险决策联动 (BR-51, BR-75, BR-69)**：
    *   在 TARA 结果表中记录 `is_human_modified: true` 和 `modification_reason`。
    *   重新分析时，若资产依赖项未变，自动继承上一轮人工结论。
    *   在阶段 ④ (风险处理决策) 为 `accept` (接受) 或 `transfer` (转移) 时，阶段 ⑤ 自动免除生成对应的 CSR。
4.  **多路径聚合与任务中止机制 (BR-67, BR-70)**：
    *   编写多路径可行性判定逻辑：取所有关联路径可行性等级的 Max 值。
    *   实现 `POST /api/domains/:domain_id/cancel-run` 中止接口，调用 `celery.control.revoke` 并重置任务进度为 0、状态为未开始。
    *   实现脱网离线手工填写表单的 fail-safe 入库机制。

### 阶段依赖
*   阶段 3 资产收集数据源正常。

### 验收标准
*   [ ] 无已确认资产时启动 TARA 分析会失败报错。
*   [ ] 运行分析时子域控状态变为只读。中途点击“取消运行”，后台任务成功终止且状态重置。
*   [ ] 对某个步骤进行人工修改并保存后，点击重新分析，如果资产信息未变，该步骤直接继承人工结论且其 Hash 校验正常。
*   [ ] 检查阶段 ④ 接受风险后的阶段 ⑤，确认未产生 CSR。

## 阶段 5：报告导出与系统配置 (第 8 - 10 周)

### 任务细分
1.  **多格式报告生成与数据脱敏 (BR-57, BR-77)**：
    *   集成 `python-docx` 与 `openpyxl` 库。
    *   实现将 TARA 结果数据与安全矩阵生成 DOCX/XLSX，输出至挂载的宿主机本地 Docker Volume。
    *   实现脱敏导出模式：在此模式下过滤隐藏底层的攻击路径、脆弱性备注，只保留 CSR 与最终风险等级。
2.  **大模型参数配置与结构化连通性测试 (BR-59, BR-71)**：
    *   提供配置保存接口（写入 `system_settings`）。
    *   测试连接时，发送包含结构化 Schema 期望的测试 Prompt，校验 LLM 返回的内容是否能被 Instructor / Pydantic 正确解析。

### 阶段依赖
*   阶段 4 的分析步骤数据输出正常。

### 验收标准
*   [ ] 导出脱敏版的 Excel 报告，确认攻击路径细节（漏洞备注、具体节点流转）均已隐藏，仅保留总风险值和 CSR 汇总。
*   [ ] 修改大模型配置为错误的 Key，连通性测试接口能正确捕捉 JSON 解析或连接超时异常并友好报错。

## 阶段 6：系统集成、安全测试与交付 (第 10 - 12 周)

### 任务细分
1.  **前后端系统级联调**：
    *   联调项目列表页模糊过滤、工作台树状导航展示。
    *   联调 React Flow 中间件自动保存与 AI 拓扑一键生成布局。
    *   测试多人在内网环境下的高并发排他锁申请与释放。
2.  **安全合规与极限用例测试**：
    *   测试 JWT 伪造、横向权限越权。
    *   测试大模型并发请求限制（Celery 并发队列限流控制）。
    *   测试极端画布数据（如孤立节点、环形依赖数据流）的资产提取与 TARA 5 阶段容错性。
3.  **系统归档冷冻测试**：
    *   验证项目变更为 `completed` 后，所有涉及 API 拦截成功。管理员账号进行解冻解锁后，各 API 读写功能恢复。

### 阶段依赖
*   所有前置开发任务就绪。

### 验收标准
*   [ ] 系统各功能在 10 人并发模拟下无死锁，乐观锁版本提示正常工作。
*   [ ] 前后端页面流畅，大模型通信延迟有 Loading 骨架屏提示。
*   [ ] 提交完整的系统交付说明、开发与运行日志记录。
