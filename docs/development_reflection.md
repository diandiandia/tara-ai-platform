## 2. 系统开发与反思

### 2.1 画布乐观锁与本地容灾测试
在模拟多用户编辑时，乐观锁能够有效避免覆盖冲突。
- **409 Conflict 体验**：当冲突发生时，前端会弹出红色模态框并提醒用户“刷新重新加载”。同时本地数据已经默默存入了 `LocalStorage` 中的 `tara_backup_conflict_<id>` 键，确保用户的设计资产绝对不会因为抢先提交而丢失。
- **网络中断测试**：一旦前端触发 `offline` 事件，系统将自动进入脱网离线模式，此时停止对 `/api/diagrams/{id}` 的直接调用，而将所有画布拓扑和版本信息写入 `tara_offline_diagram_<id>`。一旦 `online` 事件触发，系统会自动调用 `syncOfflineData` 进行原子对齐，整个流程极其顺滑。

### 2.2 独占式编辑锁 (WebSocket) 与只读机制
通过结合 Redis Key-Value (5分钟租约) 与 WebSocket 广播，当 A 用户打开画布时，该 DFD 房间内的其他所有用户都能立刻在顶部状态栏看到“只读状态 (用户 [A] 正在编辑)”，且所有画布的拖拽、添加节点以及表单提交均会被 `disabled`，阻止了并发编辑引起的逻辑覆盖。

### 2.3 5 阶段联动与风险免除
经测试，在“项目级控制矩阵”以及“TARA 5阶段审阅表”中，一旦安全专家将 Step 4 的决策更改为 `Accept` (接受) 或 `Transfer` (转移)，Step 5 的数据在后端和前端同步联动为 “免除安全需求” 并附带免除原因，确保安全目标与开发控制项的一致性。

### 3.2 待交付给人类测试的建议用例
当前项目已达交付标准。建议人类测试人员通过以下用例进行细致检验：
1.  **内网多用户排他锁抢占**：在两个不同浏览器以不同账号登录同一项目下的同一个 DFD。确认后进入的浏览器是否直接显示只读态并提示先进入者的用户名。
2.  **大模型故障降级填报**：关闭大模型连接（或故意填入错误 Key 导致 `settings/test-connection` 报错），使用“手工故障备用录入”输入各阶段数据，检查是否能正常归档并生成脱敏报表。
3.  **大项目归档冷冻审计**：以管理员身份将已完成项目进行“解冻”，以分析员身份修改部分结论，再次以管理员身份“归档”，确认分析员无法再点击 any 保存或修改按钮。
